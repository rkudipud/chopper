Overview
The construction flows contains features that are used only for specific design styles (for example RP, ctmesh, DOP). The features need to be enabled for 
some projects/designs and disabled for others. The usual method to achieve this is to enable the features by using ivars and if/else statements. This leads 
to complex code that could overwhelmed the users. 
The goal of the flow feature management is to provide a method to dynamically customize the flow to meet the design methodology while keeping the code 
simple.

Steps can be defined as:
TCL scripts
Flow builder searches for the script in the search path and adds it with full path. 
If no script is found, flow builder fails
Example:  " " step_load.tcl
Translated to:  iproc_source -file $ward/global/snps/apr_fc/step_load.tcl
ivars
In user mode, flow builder will search for the file referenced by the ivar and the file found will be added with full path.
If no script is found, flow builder fails

If the script is not found, an info message is printed and the run script will not have the step included
Example: " " setup_for_power_mesh_user.tcl -optional

Flow builder will keep the content as is

Feature Management
A feature is an  advanced flow functionality which can be added to the   basicflow. If a feature is not enabled, the code will not be visible in the flow or  
executed during the run.
Each feature requires associated step files available in the cheetah search path.
Feature Integration
Feature integration is directed by a  file named  available in the search path. For example, . json <feature_name>.feature.json csi.feature.json
The feature json file content:
---PAGE---
"name" - List of stages/steps to be processed.
Can be an identifier when the step/stage needs multiple actions. "items" field must be used in this case.
"description"  - Short description of the feature
Json format only allows one line. Use \n if necessary.
"wiki" - Web address for detailed documentation
Optional item
"target_flows"  - The feature can be applied only to specified flow - since 2025.12
Optional item
"language"  - Set to  in case of feature applied to Python based stage  - used for Python based stage"python"
"packages"  - List of Python packages used by the feature  - used for Python based stage
Each item in the name list must have a dictionary with the following fields
"action" - shows how the item should be processed. Supported actions:  add_step_before, add_step_after, add_stage_before, 
   add_stage_after, remove_step, remove_stage, load_from
"stage" - stage where the step should be added if "action" is " " add_step_*
"reference" - Step or stage used as reference for " " actionsadd_*
"flow" - Flow where the action will be implemented. Optional entry used in case multiple flows use the feature but not all items are 
wanted. Default (not used): all flows are enabled
"items" - List of steps to be inserted instead of "name". All the steps in the "item" will be inserted after/before the reference in the order 
they are listed. Only works for "add_step_*" and "remove_step" actions
When the reference step is instantiated more than once in a stage, the instance can be chosen by using  at the end of the reference string.@n
For example,  uses the second call of  as the reference. \@1 "reference": "step_load_post_compile_constraints.tcl@2" step_load_post_compile_constraints
is the same as the regular usage. If the number is bigger than the number of calls in the stage, Flow Builder will print an error and ignore the item.
An "add_stage_<>" entry can also have the following fields:
" " - Specifies the input data source /parent stage load_from
" " - Specifies the parent stage(s) when is different than "load_from" dependencies
" " - Overrides the global command for the added stage.  command
" " - List of allowed exit codes. Translated to   in the stack file. exit_codes L <exit_codes>
"" - List of output files. Translated to  in the stack file. outputs O <outputs>
"" - List of input files. Translated to I  in the stack file. inputs  <inputs>
"run_mode"  - Translated into " " in the stack file - since 2025.12 R <value>
The stage fields can be overwritten by setting "action" to the field name and 'reference" to the new value.
Example of how to change the "load_from" value of a stage.
tech.feature.json
{
    "name": [
        "redefine",
        "setup_timing",
        "initial_map"
    ],
    "description": "RTL-based flow for moving setup timing and after read_upf",
    "redefine": {
        "action": "load_from",
        "reference": "setup_timing"
    },
    "setup_timing": {
        "action": "load_from",
        "reference": "read_upf"
    },
    "initial_map": {
        "action": "load_from",
        "reference": "init_floorplan"
    }
}

Flow Configuration
The list of features to be applied can be defined in <>.features.config files. Flow builder supports 3 levels: design_class.features.config, project.features.
config, and features.config.
The files are read in the order they are listed above and the content is merged. 
2025.09.plus format
The file contains the a list of features separated by spaces and/or new line.
Still supported for 2025.12. If "features " is detected, the new format is assumed._
Since 2025.12
Syntax:
features_<flow_name> = feature1 feature2 \
feature3 ...
The list of features can be extended to multiple lines by using \. 
If a flow has no entry, no features will be applied. Empty lines and spaces are ignored.
---PAGE---
Example of configuration file
features.config
priority_apr_fc = tech rtl_based_flow mlmp mlvr tep \
                  ir_em_analysis add_redundant_vias custom_htree
priority_apr_fc_phys_block = feature1
Feature Priority
It is possible that features use the same step as reference. In this case the flow sequence depends on the order in which features are applied.
To avoid this situation, the feature priority file ( ) must be used to specify the order to avoid non-determinism. feature_priority.config
2025.09.plus format
The file contains the a list of features (one per line). Flow builder will apply the features in the order they are found in the file. The features not 
found in the file will be applied subsequently.
Still supported for 2025.12. If "priority " is detected, the new format is assumed._
Since 2025.12
Syntax
priority_<flow_name> = feature1 feature2 \
feature3 ...
The list of features can be extended to multiple lines by using \. The features will be applied in the order they are defined.
If a flow has no entry, no priority list is considered. Empty lines and spaces are ignored.
Example priority file
feature_priority.config
priority_apr_fc = tech rtl_based_flow mlmp mlvr tep \
                  ir_em_analysis add_redundant_vias custom_htree
priority_apr_fc_phys_block = feature1
The steps of a feature with lowest priority (applied later) will be closer to the reference steps that the features with higher priority.
